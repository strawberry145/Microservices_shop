# Project Brief: Women's Shoe Store — Full-Stack Microservices E-Commerce Platform

## Overview
Build a production-ready, full-stack e-commerce web application specialising in **women's footwear**. The platform must support a rich product catalogue with shoe-specific attributes (size, colour, heel height, style), a seamless shopping experience, and a robust microservices backend — all written in TypeScript.

---

## Brand & Domain Context

- **Store name:** To be defined (placeholder: `SoleHer`)
- **Target audience:** Women shopping for casual, formal, athletic, and luxury footwear
- **Key product attributes:** Size (EU/US/UK), colour, material, heel height (flat / low / mid / high / platform), style (sneakers / heels / boots / sandals / loafers / flats)
- **Business rules:**
  - Sizes are tracked individually per SKU (a size 38 red heel is a separate stock unit from size 39)
  - Out-of-stock sizes are shown but disabled on the product page
  - Low stock (≤ 3 units) triggers a "Only X left" badge
  - Orders cannot be placed for out-of-stock sizes

## Service Responsibilities

### API Gateway
- Exposes a **GraphQL API** (`apollo-server-express`) for all client-facing queries and mutations
- Exposes **REST endpoints** for health checks, webhooks, and non-GraphQL operations
- Routes all internal traffic to microservices via **gRPC**
- Handles **JWT authentication** and request validation using `zod`
- Applies rate limiting on checkout and auth endpoints

### Microservice 1 — Order Service
- Manages the full order lifecycle: creation, status updates, history, and cancellation
- Enforces stock validation before confirming any order
- Persists data in **MySQL** (`orders`, `order_items`, `users`, `payments` tables)
- Publishes Kafka events: `order.placed`, `order.confirmed`, `order.cancelled`, `order.shipped`, `order.delivered`

### Microservice 2 — Inventory Service
- Manages the full product catalogue: shoe listings, SKUs per size/colour, pricing, categories, and images
- Tracks stock per SKU (size + colour combination)
- Consumes Kafka events from the Order Service to decrement stock in real time
- Publishes Kafka events: `stock.updated`, `stock.low`, `stock.out`, `product.created`, `product.updated`

### Microservice 3 — Notification Service
- Listens to Kafka events from Order and Inventory services
- Sends transactional emails for: order confirmation, shipping update, delivery confirmation, low-stock admin alert
- Persists all notification logs (recipient, type, timestamp, status) to a **JSON store** using `lowdb`

## Kafka Event Contracts

| Topic | Producer | Consumers | Payload |
|---|---|---|---|
| `order.placed` | Order Service | Inventory, Notification | `{ orderId, userId, items[{ skuId, quantity, unitPrice }], total }` |
| `order.confirmed` | Order Service | Notification | `{ orderId, userId, estimatedDelivery }` |
| `order.shipped` | Order Service | Notification | `{ orderId, trackingNumber, carrier }` |
| `order.delivered` | Order Service | Notification | `{ orderId, deliveredAt }` |
| `order.cancelled` | Order Service | Inventory, Notification | `{ orderId, items[{ skuId, quantity }] }` |
| `stock.updated` | Inventory Service | — | `{ skuId, productId, newStock }` |
| `stock.low` | Inventory Service | Notification | `{ skuId, productId, size, colour, currentStock }` |
| `stock.out` | Inventory Service | Notification | `{ skuId, productId, size, colour }` |

## Database Schema (MySQL)

```sql
-- Users
CREATE TABLE users (
  id VARCHAR(36) PRIMARY KEY,
  email VARCHAR(255) UNIQUE NOT NULL,
  password_hash VARCHAR(255) NOT NULL,
  full_name VARCHAR(255),
  phone VARCHAR(20),
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Addresses
CREATE TABLE addresses (
  id VARCHAR(36) PRIMARY KEY,
  user_id VARCHAR(36) REFERENCES users(id),
  label VARCHAR(50),
  street TEXT NOT NULL,
  city VARCHAR(100) NOT NULL,
  country VARCHAR(100) NOT NULL,
  postal_code VARCHAR(20) NOT NULL,
  is_default BOOLEAN DEFAULT FALSE
);

-- Products (one row per shoe model)
CREATE TABLE products (
  id VARCHAR(36) PRIMARY KEY,
  name VARCHAR(255) NOT NULL,
  description TEXT,
  brand VARCHAR(100),
  style ENUM('sneakers','heels','boots','sandals','loafers','flats','platforms','mules'),
  heel_height ENUM('flat','low','mid','high','platform'),
  material VARCHAR(100),
  base_price DECIMAL(10,2) NOT NULL,
  category VARCHAR(100),
  is_active BOOLEAN DEFAULT TRUE,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Product Images
CREATE TABLE product_images (
  id VARCHAR(36) PRIMARY KEY,
  product_id VARCHAR(36) REFERENCES products(id),
  url VARCHAR(500) NOT NULL,
  alt_text VARCHAR(255),
  is_primary BOOLEAN DEFAULT FALSE,
  sort_order INT DEFAULT 0
);

-- SKUs (one row per size + colour combination)
CREATE TABLE skus (
  id VARCHAR(36) PRIMARY KEY,
  product_id VARCHAR(36) REFERENCES products(id),
  size_eu DECIMAL(4,1) NOT NULL,
  size_us DECIMAL(4,1),
  size_uk DECIMAL(4,1),
  colour VARCHAR(50) NOT NULL,
  colour_hex VARCHAR(7),
  stock_quantity INT DEFAULT 0,
  price_override DECIMAL(10,2),
  sku_code VARCHAR(100) UNIQUE NOT NULL
);

-- Orders
CREATE TABLE orders (
  id VARCHAR(36) PRIMARY KEY,
  user_id VARCHAR(36) REFERENCES users(id),
  status ENUM('pending','confirmed','shipped','delivered','cancelled'),
  total DECIMAL(10,2) NOT NULL,
  shipping_address_id VARCHAR(36) REFERENCES addresses(id),
  tracking_number VARCHAR(100),
  carrier VARCHAR(100),
  notes TEXT,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- Order Items
CREATE TABLE order_items (
  id VARCHAR(36) PRIMARY KEY,
  order_id VARCHAR(36) REFERENCES orders(id),
  sku_id VARCHAR(36) REFERENCES skus(id),
  quantity INT NOT NULL,
  unit_price DECIMAL(10,2) NOT NULL
);

-- Wishlist
CREATE TABLE wishlist_items (
  id VARCHAR(36) PRIMARY KEY,
  user_id VARCHAR(36) REFERENCES users(id),
  product_id VARCHAR(36) REFERENCES products(id),
  added_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

## End-to-End Flow to Implement

> Customer browses shoe catalogue → filters by size 38 and heel height "high" → opens product detail → selects colour and size → adds to cart → proceeds to checkout → submits order → Order Service validates stock per SKU → saves order in MySQL → publishes `order.placed` to Kafka → Inventory Service decrements stock for that SKU → if stock drops to ≤ 3, publishes `stock.low` → Notification Service sends order confirmation email and logs it → client shows order confirmation page with estimated delivery.
